# シグナル探索ガイド

`SKILL.md` の各分析軸について、**理論的背景・計算式の留意点・実務での読み方** を補足する。
モデルが計算ミスをしないことと、結果を解釈にどう翻訳するかを両方カバーする。

## 1. なぜ「予測力スクリーニング」レイヤーが必要か

シグナル候補を「使える」と判断する前に、最低限以下を確認したい:

1. シグナル値と将来リターンに **そもそも相関があるか**
2. 単純に符号を見ているだけで PnL が出るか
3. 既存指標と **冗長でないか**（同じ情報を別名で再発見していないか）
4. **リーク** していないか（未来情報を参照していないか）
5. 値の分布が変動して相関が安定しないだけで、**変換すれば改善するか**

これを最適化（閾値スイープ、コスト調整、サイジング）の前に通す層が
「シグナル探索」。最適化を先にやると、ノイズに合わせ込んだ「good story」が
できあがってしまい、実運用で必ず崩れる（Vol.04 / Vol.06 の教訓）。

## 2. 各分析軸の理論的背景

### 2.1 生データ vs 将来リターン相関

将来リターン:

```
r_fwd[t] = log(close[t+h] / close[t])      # h はデフォルト 1
```

- **対数リターンを使う理由:** 加法性。複数足合計の議論がしやすい
- **horizon h:** デフォルト 1（次足）。ユーザー指定で 24 や 168 も可。
  ただし h を伸ばすほどサンプル数が減り、ファットテールの影響で
  相関の標本誤差が膨らむ

**Pearson r vs Spearman r:**

| 比較 | 含意 |
|-----|------|
| \|Pearson - Spearman\| < 0.05 | ほぼ線形。シンプルなモデルで十分 |
| Pearson > Spearman に大きく振れる | 外れ値が Pearson を押し上げ。ロバスト変換（log、Z-score、winsorize）の余地 |
| Spearman > Pearson に大きく振れる | 順位連動はあるが線形には乗らない。非線形変換を検討 |

**p 値の扱い（重要）:**

- 暗号資産時系列は **i.i.d. でない**（自己相関 + ファットテール）
- 古典的 Pearson の p 値は i.i.d. 前提なので **過小評価**（楽観バイアス）
- 「N が大きいので p < 0.001 です！」という結論は鵜呑みにしない
- 本 skill は **r の絶対値とローリング相関の安定性を優先** し、
  p 値は補助情報として表示するのみ

### 2.2 ローリング Z-score 変換

```
z[t] = (s[t] - rolling_mean(s, w)[t]) / rolling_std(s, w)[t]
```

**なぜローリングか:**

- グローバル正規化（全期間の mean/std で割る）は **未来情報リーク**
- レジーム変化（強気 → 弱気）でシグナルの平均値・分散が動く場合、
  グローバル正規化では「直近のシグナル値が高い／低い」の情報が薄まる
- ローリングなら直近 w 本での相対位置だけで判断 → 定常化

**改善が大きい候補:**

- **非定常な水準そのもの**（出来高、価格、プレミアム）
- **トレンド成分が乗った指標**（SMA そのもの、close - SMA の絶対値）

**改善が小さい候補:**

- **元から定常な指標**（RSI、Stochastic、ROC）
- これらは設計時点で 0〜100 や ±X% にバウンドされているので、
  Z-score 化してもさほど変わらない

### 2.3 ラグ相関プロファイル

```
corr_lag[k] = Pearson(s[t], log(close[t+k] / close[t+k-1]))   for k = 1..K
```

**読み方:**

- **lag=+1 にピーク** → 即時反応のシグナル（次足だけ効く）。
  約定タイミングがシビア。エントリーの遅延に弱い
- **lag=+5〜+10 にピーク** → 中期反応。エントリーが多少遅れても効く
- **どこにもピークがなく一様にゼロ** → ノイズ。捨てる

**多重比較問題:**

- 48 ラグを同時に検定すると、α = 0.05 で **約 91% の確率** で 1 つは
  「有意」になる（偶然）
- Bonferroni 補正: α / K = 0.05 / 48 ≈ 0.001 を使う
- 厳しすぎる場合は **FDR 補正（Benjamini-Hochberg）** を使う
- 本 skill は補正後の p 値は出さないが、ピーク 1 個に飛びつく前に
  「48 個も見ているのだから当たってもおかしくない」という視点を必ず添える

### 2.4 自己相関（粘着性）

```
acf[k] = Pearson(s[t], s[t-k])
```

**用途別の指標分類:**

| 自己相関の特徴 | 代表的な指標 | 適した用途 |
|---------------|------------|-----------|
| lag=1 で > 0.95、lag=24 でも > 0.5 | ATR、SMA そのもの、ボラ系 | **レジーム検出**。「今は高ボラ局面か」 |
| lag=1 で 0.7〜0.95、lag=24 で 0.1〜0.3 | RSI、Stoch、MACD ヒスト | レンジ判定・ダイバージェンス |
| lag=1 で < 0.3 | リターン、価格差分系 | **エントリー信号**。即応性が必要な用途 |
| lag=1 で > 0.99 | 価格そのもの、長期 SMA | **疑え**。シグナルが価格と同義になっていないか |

**注意:** 自己相関が高すぎる（粘着的）シグナルでエントリーすると、
**同じシグナル値が長期間続くため過剰トレード or 過少トレード** になる。
レジーム情報として使うか、差分（ROC 化）してから使う。

### 2.5 既存指標との冗長性

カテゴリ別に最低 1 つずつ既存指標を選ぶ:

| カテゴリ | 代表指標 |
|---------|---------|
| トレンド | SMA(20) 乖離率、EMA(20) 乖離率 |
| モメンタム | RSI(14)、ROC(12) |
| ボラ | ATR(14)/close、BB 幅(20) |

**判定:**

- |r| > 0.8 → **冗長**。既存指標で代替可能なので採用メリット少
- |r| < 0.3（特に異なるカテゴリと） → **独立**。組み合わせで価値が出る可能性
- 評価対象シグナル自身を既存セットに混ぜない（自己相関 1.0 のフラグ漏れ）

**実務的な使い方:**

- 評価対象が **既存ボラ系と独立** で、既存モメンタム系と冗長 → 既存ボラと組み合わせ
- 評価対象が **どれとも独立** → 新しい次元の情報源かもしれない（要 backtest）
- 評価対象が **複数の既存指標と冗長** → 別名の同じもの。スキップ推奨

### 2.6 符号ベース簡易 PnL

```
position[t] = sign(s[t])           # +1 / -1 / 0
pnl[t]      = position[t-1] * r[t] # shift(1) で次足始値約定相当
cum_pnl[t]  = sum(pnl[0..t])
```

**設計思想:**

- **コスト無視**: 手数料 0.12% × 往復 = 0.24% のドラッグを完全に外す。
  予測力の純粋なシグナル / ノイズ比を見る
- **閾値最適化なし**: シグナルの符号だけで判断。「+1σ 超えたら買い」のような
  カットオフは恣意的になりがちなので本 skill では使わない
- **shift(1) 厳守**: t のシグナルで t+1 の始値約定相当。
  リーク自己チェック（Step 7）以外で shift を外さない

**ショートサイドの扱い:**

- bitbank 現物では **ロングのみ実行可能**
- 簡易 PnL のショート（sign < 0）は **方向検証用の数学シミュレーション**
- 「シグナルがマイナスのとき下落するか」を確認するための数値計算であり、
  実運用に直結しない

**Buy & Hold との比較:**

- 簡易 PnL が Buy & Hold を **大幅に下回る** のが普通（コスト無視でも）
- それでもシグナルとして意味があるのは「ドローダウン制御」「ヘッジ判断」等の
  別軸の価値。本格評価は backtest skill で

### 2.7 リーク自己チェック

backtest skill の Step 3.5 と同じ哲学:

- shift(1) **あり版**（正規）と **なし版**（リーク版）を同条件で計算
- 差が極端に大きい → リーク疑い

**リークの典型例:**

1. シグナル計算で `t` 時点の close を使い、約定も `t` の close で行う（同足リーク）
2. rolling 計算で `center=True` を指定して未来データ参照
3. 全期間正規化（mean/std を全データで計算）
4. 経路 C の外部データで、データの timestamp が「集計完了時刻」ではなく
   「集計開始時刻」になっている（実際には未来情報）

**判定後の流れ:**

- リーク疑いなし → 「予測力検証として信頼できる」と表示し、判定ラベル付与へ
- リーク疑いあり → ロジック再確認をユーザーに促し、判定は保留

## 3. 経路 C: 外部データ入力 JSON フォーマット仕様

ユーザーが事前計算した時系列を渡すための汎用フォーマット。
将来 `correlation-analysis` skill でも使う想定で設計。

### 3.1 最小フォーマット

```json
{
  "name": "coinbase_premium",
  "timestamps": [1712025600000, 1712029200000, 1712032800000],
  "values": [0.0012, -0.0035, 0.0008]
}
```

- `name`: シグナル名（出力ラベルに使用）
- `timestamps`: UTC ミリ秒の配列（bitbank candles と同形式）
- `values`: 数値の配列。`null` / NaN が混じる場合は除外する

### 3.2 拡張フォーマット（任意）

```json
{
  "name": "coinbase_premium",
  "description": "Coinbase BTC/USD - Bitstamp BTC/USD price ratio - 1",
  "source": "external",
  "timestamps": [1712025600000, ...],
  "values": [0.0012, ...],
  "meta": {
    "timezone": "UTC",
    "interval_ms": 3600000,
    "preprocessing": "log-ratio, no normalization"
  }
}
```

`meta` は出力に併記される情報メタデータ。計算には影響しない。

### 3.3 candles との結合手順

1. CLI で candles を `--format=json` で取得
2. candles の各行から `timestamp_ms` を抽出
3. 入力 JSON と **left-join**（candles を左、外部データを右）
4. 外部データ側が NaN または欠損の行は **除外**（0 埋めしない）
5. 結合後の `s[t]`、`r_fwd[t]` で Step 1〜7 を実行

### 3.4 制約と注意

- **bitbank CLI で取得不可な外部データはこの経路で受け取る**
  - 例: Coinbase / Bitstamp / Binance の価格、海外の出来高、オンチェーン指標
- **本 skill が外部 API を直接叩くことはしない**（CLAUDE.md のスコープ）
- **timestamp の解釈ズレに注意**: 集計完了時刻 vs 集計開始時刻。
  外部データが「12:00 の値」というとき、12:00 〜 13:00 の集計か、
  11:00 〜 12:00 の集計かでリーク方向が変わる
- **interval mismatch**: candles が 1hour で外部データが 1day のとき、
  forward-fill するか、resample するかをユーザーに確認する

## 4. 多重比較・p 値の扱い

### 4.1 なぜ p 値を強調しないか

- 暗号資産時系列は i.i.d. でない（自己相関、ファットテール、レジーム変化）
- 古典的 p 値の前提が崩れているので、**サンプル数の多さがそのまま「有意」を
  生む方向のバイアス** がある
- N=6,576 で r=0.02 でも p < 0.001 が出る。だが r=0.02 は実用的にはノイズ
- **r の絶対値とローリング相関の符号安定性** の方が判断材料として頑健

### 4.2 多重比較が必要な場面

- ラグ相関で 48 ラグを同時に見るとき
- 複数のシグナル候補を一括スクリーニングするとき
- 期間を切り替えて「どこかで効くか」を探すとき

### 4.3 補正の選択肢

| 補正 | 計算 | 性質 |
|-----|------|------|
| Bonferroni | α / K | 厳しい。偽陽性最小化 |
| Holm-Bonferroni | 順序付き α / (K - i + 1) | Bonferroni より少し緩い |
| Benjamini-Hochberg (FDR) | 偽陽性割合の制御 | 多検定向け。実務寄り |

本 skill では補正後 p 値を自動計算しないが、ユーザーが「Bonferroni 補正して」と
指示すれば追加計算する。

### 4.4 実務的な代替

p 値補正の代わりに以下を重視する:

- **ローリング相関**: window=720（1 ヶ月相当 1hour）で時系列に相関を計算し、
  符号がどれだけ安定しているかを見る。頻繁に符号反転するなら本物の相関ではない
- **アウトオブサンプル**: 期間を半分に割って、前半でフィットした閾値が
  後半でも機能するかを backtest で確認

## 5. promising 判定後の進め方

`✅ promising` が出たシグナルを **そのまま実運用に投入してはいけない**。
本 skill はあくまで **スクリーニング層**。次の段階での検証が必須。

### 5.1 推奨フロー

1. **本 skill で promising 判定**
2. **backtest skill で戦略化**:
   - 閾値（buy if z > +1.0, sell if z < -1.0 等）を決める
   - 手数料 0.12% × 往復を入れる
   - イグジット条件（逆閾値、トレーリングストップ、時間切れ）を定義
3. **In-Sample / Out-of-Sample 分割**:
   - 期間を 70/30 等で分割
   - 前半で閾値最適化、後半で検証
4. **パラメータ感度分析**:
   - 閾値を ±20% 動かしても PnL が崩れないか
   - 1 点の閾値でだけ機能するなら過適合疑い
5. **ウォークフォワード検証**:
   - 月次で再パラメータ化して連結
6. **実運用前の Paper Trade 期間**

### 5.2 パラメータ最適化の落とし穴

- **In-Sample で最適化、Out-of-Sample で検証** を分離
- 多数のパラメータ組み合わせを試すと **必ず偶然性能が良いものが出る**
  （多重比較と同じ問題）
- 「最適パラメータ周辺で性能が崩れない（プラトーがある）」かを必ず確認
- 1 点だけ突出しているチャンピオンは過適合

詳細は `bitbank-lab/crypto-data-analysis-course/vol08*` 系の教材を参照。

## 6. 各分析軸の典型値（BTC/JPY 1hour, 過去 1 年）

Validation Loop で「明らかに変な値」を検出するための目安。

| 項目 | 想定範囲 | 範囲外なら疑うこと |
|-----|---------|------------------|
| RSI(14) と r_fwd の Pearson r | -0.02 〜 +0.02 | \|r\| > 0.05 → リーク or 過去データ重複 |
| RSI(14) の自己相関 lag=1 | 0.85 〜 0.95 | < 0.5 → 期間設定ミス、> 0.99 → 価格そのものに近い |
| SMA(20) 乖離率 と RSI(14) の r | 0.7 〜 0.9 | < 0.5 → どちらかの計算ミス |
| ATR(14)/close の自己相関 lag=1 | 0.95 〜 0.99 | < 0.8 → ATR の計算ミス |
| BB 幅(20) と ATR/close の r | 0.5 〜 0.8 | 大きく外れる → どちらかの計算ミス |
| 簡易 PnL（生データ符号、コスト無視） | -10% 〜 +10% | 絶対値が 30% 超 → リーク疑い |
| shift なし PnL / shift あり PnL | 0.5x 〜 2.0x | 2.0x 超 → leak-suspect |
